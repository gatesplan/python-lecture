# 1-3 탐색 알고리즘

## 우리는 왜 탐색(Search)을 할까?

- 시간에 민감한 작업: 경로를 이탈했을 때 내비게이션은 1초라도 빨리 새로운 목적지 경로를 찾아야 한다.
- 효율성: 수십억 개의 웹 페이지 속에서 내가 원하는 정보를 단 몇 초 만에 정확히 찾아낸다.
- 방대한 데이터 속에서 정보를 빠르고 정확하게 찾는 기술이 바로 탐색 알고리즘이다.

## 데이터의 지도, '데이터 구조'

- 데이터 구조는 선형 데이터 구조와 비선형 데이터 구조로 나뉜다.
- 선형 데이터 구조: 배열, 리스트, 스택, 큐.
- 비선형 데이터 구조: 그래프, 트리.
- 오늘의 목표: 한 줄로 나열된 선형 데이터 구조 안에서 원하는 값 찾기.

## 선형 vs 비선형 데이터 구조

선형 데이터 구조
- 요소들이 차례로 연결되는 순차적 방식(1:1 구조).
- 구현이 쉽고 특정 요소에 직접 접근 가능(배열 등).
- 적용 알고리즘: 순차 탐색, 이진 탐색.

비선형 데이터 구조
- 한 요소가 여러 요소와 연결된 계층/관계형(1:n, n:n 구조).
- 특정 노드에 한 번에 접근 불가.
- 적용 알고리즘: 깊이 우선 탐색, 너비 우선 탐색.

## 무식하지만 확실한 방법: 순차 탐색 (Sequential Search)

- 정의: 리스트의 첫 번째 요소부터 마지막까지 순서대로 하나씩 확인하며 원하는 값을 찾는 알고리즘.
- 특징: 도서관 책이 뒤죽박죽 섞여 있어도 1번 책부터 끝까지 다 찾아보면 무조건 찾을 수 있다.
- 핵심: 데이터가 정렬되어 있지 않아도 사용할 수 있다.

## 순차 탐색의 작동 원리 (목표: '5' 찾기)

배열 예시: [4, 6, 10, 2, 3, 1, 7, 5]

- 인덱스 0 검사 -> 4 (아님)
- 인덱스 1 검사 -> 6 (아님)
- ...
- 인덱스 7 검사 -> 5 (찾았다)

총 8번의 비교 끝에 탐색 성공.

## 순차 탐색의 현실 (수행 시간)

- 최선의 경우(운이 좋을 때): 찾는 값이 맨 앞에 있을 때 -> 비교 횟수 1번.
- 최악의 경우(운이 나쁠 때): 찾는 값이 맨 끝에 있거나 아예 없을 때 -> 비교 횟수 n번.
- 데이터(n)가 많아질수록 탐색 시간도 정비례하게 같이 늘어나는 O(n)의 한계를 가진다.

## 반으로 쪼개는 마법: 이진 탐색 (Binary Search)

- 정의: 리스트의 중간값을 확인하고, 찾는 값이 중간값보다 작으면 왼쪽 절반, 크면 오른쪽 절반을 탐색하는 과정을 반복하는 알고리즘.
- 절대 규칙: 데이터를 탐색하기 전에 반드시 정렬되어 있어야만 작동한다.
- 매 단계마다 탐색 범위의 50%를 제거한다.

## 이진 탐색: 데이터의 바다에서 단숨에 정답을 찾는 법

파이썬으로 구현하는 이진 탐색(Binary Search) 알고리즘 완벽 해부.

## 1부터 100 사이의 숫자를 가장 빨리 맞히는 방법은?

- 순차 탐색(바보 같은 방법): "1인가요? 2인가요? 3인가요?..."
- 이진 탐색(스마트한 방법): "50보다 큰가요?"

절반을 과감하게 버리는 것, 이것이 효율적인 탐색 알고리즘의 시작이다.

## 속도의 차이가 알고리즘의 차이를 만든다

| 탐색 방식 | 순차 탐색(Linear) | 이진 탐색(Binary) |
| --- | --- | --- |
| 작동 원리 | 처음부터 끝까지 하나씩 확인 | 반씩 쪼개며 탐색 범위 축소 |
| 100만 개 탐색 시(최악) | 1,000,000번 확인 필요 | 단 20번 확인으로 종료 |
| 필수 전제 조건 | 없음 | 정렬(Sort) |

절반을 과감하게 버리는 것, 이것이 효율적인 탐색 알고리즘의 시작이다.

## 대전제: "모든 데이터는 정렬되어 있어야 한다"

1. 정렬 전(Unsorted)
2. .sort() 적용(Sort)
3. 정렬 후(Sorted)

탐색을 시작하기 전, .sort() 함수는 선택이 아닌 필수다. 데이터가 크기순으로 정렬되어 있어야만 컴퓨터가 '어느 방향으로 갈지'를 결정할 수 있다.

## 범위를 좁히는 3개의 지시자 (Pointers)

- Left, Mid, Right 3개 포인터.
- 알고리즘의 원리는 단순하다. 오직 이 3개의 포인터를 요리조리 움직이며 탐색 범위를 좁혀나가는 게임이다.

## Action: 타겟이 중앙값(Mid)보다 큰가, 작은가?

- 비교: Target과 Mid를 비교한다.
- Target이 더 작을 때: Left는 그대로, Right = Mid - 1(오른쪽 절반 제외).
- Target이 더 클 때: Right는 그대로, Left = Mid + 1(왼쪽 절반 제외).

찾고자 하는 값이 중간값보다 크다면, 왼쪽 절반은 더 이상 볼 필요가 없다. 과감하게 버려라.

## 알고리즘을 파이썬 함수로 설계하기

- INPUTS: arr(리스트 데이터), target(찾을 값).
- 함수: `def binary_search(arr, target):`
- RETURN: mid(찾은 위치 인덱스).

2개의 재료를 입력받아 1개의 결과값을 돌려주는 (I, O) 형태의 완벽한 사용자 정의 함수를 조립한다.

## 언제까지 탐색할 것인가? 멈춰야 할 때를 아는 브레이크

- while문은 조건이 참인 동안 반복한다.
- `left <= right`란 "아직 찾아볼 구간이 남아있다"는 의미다.
- 두 포인터가 역전되는 순간(데이터가 리스트에 없음), 루프는 종료된다.

```
left = 0
right = len(arr) - 1
while left <= right:
```

## 방 번호(인덱스)는 반드시 정수여야 한다

- `(1 + 4) / 2 = 2.5` -> 에러. 소수점 방 번호는 존재하지 않는다.
- `(1 + 4) // 2 = 2` -> 성공. 소수점을 버리고 정확한 인덱스를 찾는다.
- 파이썬의 정수 나누기 연산자 `//`가 이진 탐색 코드에서 절대적으로 필요한 이유다.

## 조건문으로 탐색 범위 절반 날려버리기

- `if arr[mid] == target:` -> 정답 반환(return mid).
- `elif arr[mid] < target:` -> 오른쪽 절반 탐색(Left = mid + 1).
- `else:` -> 왼쪽 절반 탐색(Right = mid - 1).

단 3갈래의 조건 분기문이 방대한 데이터를 무자비하게 반으로 잘라내는 정밀 타격 조준경 역할을 한다.

## 코드가 실행되는 찰나의 순간 (루프 1회차)

- Target = 7, 리스트 예시.
- 첫 번째 탐색. 단 한 번의 비교로 데이터의 절반(0~4 구간)이 완전히 탐색 대상에서 제외된다.
- 초기: Left(0), Right(9), Mid(4).

## 타겟 발견 및 탐색 종료

- 단 2번의 루프만에 탐색 완료.
- 순차탐색이었다면 8번의 확인이 필요했을 작업이다.
- `if arr[mid] == target:` 조건이 참이 되면 `return mid`.

| 데이터 개수 | 순차 탐색(비교 횟수) | 이진 탐색(비교 횟수) |
| --- | --- | --- |
| 100개 | 100번 | 7번 |
| 10,000개 | 10,000번 | 14번 |
| 1,000,000개 | 1,000,000번 | 20번 |

순차 탐색: 0번 강수빈부터 하나씩 확인. 결과: 비교 횟수 8번.

이진 탐색: 중간 위치 '박민서' 확인 -> 아래 절반의 중간 '이효주' 확인. 결과: 비교 횟수 2번.

| --- | --- | --- |
| 탐색 방식 | 처음부터 하나씩 순서대로 검사 | 중간값을 기준으로 절반씩 버리며 탐색 |
| 사전 조건 | 정렬 불필요(아무렇게나 섞여 있어도 OK) | 반드시 오름차순/내림차순 정렬되어야 함 |
| 대규모 데이터 효율 | 비효율적(데이터가 많을수록 느려짐) | 매우 효율적(데이터가 방대해도 엄청 빠름) |

## 일상 속의 탐색 알고리즘

- 순차 탐색: 도서관에서 책 찾기. 책들이 순서대로 꽂혀있지 않다면? 1번 책장부터 하나씩 다 찾아야 한다.
- 이진 탐색: 사전에서 단어 찾기. 'Algorithm'을 찾을 때, 중간을 펼쳐보고 알파벳 순서에 따라 남은 절반을 탐색한다(정렬되어 있기 때문에 가능).

## 개발자의 선택: 어떤 알고리즘을 써야 할까?

- 데이터가 정렬되어 있나요?
  - 아니요 -> 순차 탐색(Sequential Search) 사용.
  - 네 -> 데이터의 양이 엄청나게 많은가요?
    - 아니요 -> 순차 탐색 사용.
    - 네 -> 이진 탐색(Binary Search) 사용.

탐색 알고리즘은 데이터의 특성(정렬 여부, 크기)에 따라 완벽한 도구를 선택하는 것이 핵심이다.

## 데이터의 바다를 탐험하는 분할 정복의 미학

자기 자신을 호출하는 마법, 재귀함수로 구현하는 이진 탐색의 완벽한 논리 구조.

## 재귀(Recursion): 거울 속의 거울을 들여다보는 논리

재귀함수는 완전히 동일한 패턴의 더 작은 문제를 해결하기 위해 '자기 자신'을 다시 호출하는 함수이다. 단순한 코딩기법이 아니라, 복잡한 현상을 가장 작은 단위로 쪼개어 바라보는 사고방식이다.

## 무한 루프를 막는 재귀함수의 두 가지 핵심 기둥

모든 재귀함수는 무한히 빠져드는 것을 방지하기 위해 두 가지 필수 요소를 가져야 한다. 이 두 가지가 결합될 때 비로소 재귀는 '작동'한다.

- 종료 조건(Base Case): 더 이상 쪼갤 수 없을 때 함수를 멈추는 명확한 기준.
- 재귀 호출(Recursive Step): 문제를 한 단계 더 작게 만들어 자기 자신을 다시 부르는 과정.

## (재귀 이진 탐색 코드)

코드는 `binary_search_recursive.py` 참고.

## 재귀적 이진 탐색의 첫 번째 단계: 중앙값 비교

배열의 정중앙을 확인한다. 우리가 찾는 값(Target)이 중앙값보다 크다면, 왼쪽 절반은 더 이상 탐색할 필요가 없다. 이제 오른쪽 절반을 대상으로 '자기 자신'을 호출한다.

예: 배열 [1, 3, 5, 7, 9, 11, 13], Target: 11, Low(0), Mid(3), High(6) -> Target(11) > Mid(7).

## 새로운 범위에서의 재귀 호출과 탐색 종료

업데이트된 새로운 경계(Low = Mid + 1)를 가지고 함수가 재귀적으로 실행된다. 중앙값이 우리가 찾는 값과 일치하는 순간, 탐색은 즉시 종료(Base Case)된다.

예: 새 범위 [9, 11, 13], Low, Mid, High -> Target(11) == Mid(11) -> 정답 발견(Base Case).

## 인간의 논리를 그대로 닮은 파이썬 코드 구조

복잡한 반복문 없이, 오직 순수한 논리적 조건만으로 이루어진 깔끔한 코드 아키텍처.

- `if low > high: return -1` -> 종료 조건 1: 탐색 실패(Base Case).
- `if arr[mid] == target: return mid` -> 종료 조건 2: 탐색 성공(Base Case).
- `return binary_search_recursive(...)` -> 재귀 호출: 범위를 좁혀 자기 자신을 다시 호출.

## 컴퓨터는 정답을 어떻게 위로 전달하는가?

정답을 찾았다고 끝나는 것이 아니다. 메모리 깊은 곳에서 찾은 인덱스 값은 콜 스택을 타고 역순으로 호출자에게 연쇄적으로 전달되어야 비로소 최종 결과가 된다.

예시 흐름:
- `search(arr, target=11, low=0, high=6)` [대기 중...]
- `search(arr, target=11, low=4, high=6)` [인덱스 5 반환]
- Index 5가 상위 호출로 전달됨.

## 반복문(While)과 재귀함수(Recursive)의 구조적 차이

두 방식 모두 동일한 이진 탐색을 수행하지만, 코드를 설계하는 철학과 메모리를 사용하는 방식에서 명확한 차이를 보인다.

| 구분 | 반복문 방식(Iterative) | 재귀함수 방식(Recursive) |
| --- | --- | --- |
| 논리 전개 / Logical Flow | 상태(변수)를 갱신하며 루프를 도는 구조 | 더 작은 문제로 쪼개어 자기 자신에게 맡기는 구조 |
| 메모리 효율 / Memory | 메모리 사용량이 적음(콜 스택 오버헤드 없음) | 재귀 깊이가 깊어질수록 콜 스택 메모리 사용량 증가 |
| 코드 가독성 / Readability | 루프 조건과 상태 변화를 추적해야 함 | 압도적으로 우수함. 수학적 정의와 인간의 논리를 코드에 그대로 매핑 |

## 1-3 탐색 알고리즘 핵심 요약

- 방대한 데이터 시대, 빠른 탐색 알고리즘은 현대 기술의 필수 엔진이다.
- 순차 탐색은 유연하지만 데이터가 많아질수록 느려지는 O(n)의 한계가 있다.
- 이진 탐색은 절반씩 탐색 범위를 줄여 압도적으로 빠르지만, 반드시 '정렬'이라는 대가가 필요하다.
