## 2번

정답 예시:

```python
arr = [4, 6, 10, 2, 3, 1, 7, 5]

def search(x):
    for i in range(len(arr)):
        if arr[i] == x:
            return i
    return -1


print(search(7))
```

채점 포인트:

- 인덱스 순회 + 일치 시 그 인덱스 반환 + 끝까지 못 찾으면 `-1` 반환, 이 세 가지가 모두 들어가 있어야 만점.
- 일치하는 순간 `return i`로 즉시 빠져나오는지 확인. (불필요하게 끝까지 루프를 도는 구조는 의미상 동작은 하지만 "순차 탐색"의 본질을 어긴 것으로 본다.)
- `return -1`이 반복문 **바깥**에 위치해야 한다. 들여쓰기를 잘못 잡아 `for` 안쪽에 넣으면 첫 비교에서 무조건 `-1`을 반환하므로 오답.

감점 사유:

- **길이 하드코딩**: `range(len(arr))` 대신 `range(8)`로 작성. 동작은 하지만 리스트 길이가 바뀌면 깨지는 코드이므로 감점. 출제 의도(`len()`을 통한 일반화)에 어긋남.
- **반환값 누락**: 함수가 값을 `return`하지 않고 `print(i)`로 출력만 하거나, 못 찾았을 때 아무것도 반환하지 않음. 함수 시그니처 요건 미충족.
- **불필요한 변수**: 별도의 `found` 플래그 변수를 만들어 끝까지 순회한 뒤 반환하는 식의 비효율적 구현. 만점은 어려움(부분 점수).
- **`while`로 작성**: 출제 의도는 `for`이지만 `while`로 동일 동작을 구현했다면 동작 정확성에 한해 만점 인정 가능 (출제 의도와 일치 여부는 출제자 판단).

## 3번

채점 항목은 두 가지로 나누어 합산한다. 만점 = A + B = 100%.

### A. 정렬 전제 조건 언급 (50%)

리스트(데이터)가 **정렬되어 있어야** 이진 탐색을 적용할 수 있다는 사실을 명시했는가.

- 만점(50%): "정렬되어 있어야 한다", "오름차순/내림차순으로 나열되어 있어야 한다" 등 명확한 표현.
- 0%: 정렬에 대한 언급이 전혀 없음.

### B. 절반씩 버리는 메커니즘 서술 (단계별 부분 점수, 최대 50%)

탐색 범위를 어떻게 좁혀가는지에 대한 서술의 디테일에 따라 두 단계로 채점.

- **단순 수준 (20%)**: "반으로 나누어가며 찾는다" 정도의 서술. 절반씩 줄어든다는 아이디어는 인지하지만 어느 쪽을 버리는지의 기준은 빠져 있음.
- **상세 수준 (50%)**: 중간값(`mid`)을 찾는 값과 **비교**해서, 찾는 값이 더 크면 오른쪽 절반만 / 더 작으면 왼쪽 절반만 남기고 나머지 절반을 버린다는 메커니즘까지 명시. 비교 기준과 버리는 방향이 함께 들어가 있어야 만점.
- 0%: 절반씩 줄인다는 아이디어 자체가 없음.

## 4번

정답:

- (가): `arr[mid] > target`
- (나): `binary_search(arr, target, low, mid - 1)`
- (다): `binary_search(arr, target, mid + 1, high)`

대안 정답: (가)에 `arr[mid] < target`을 쓴 경우, (나)와 (다)의 재귀 호출이 서로 바뀌어야 함.

- (가) `arr[mid] < target` → (나) `binary_search(arr, target, mid + 1, high)`, (다) `binary_search(arr, target, low, mid - 1)`

배점 (총 100%):

| 항목 | 배점 | 비고 |
|---|---|---|
| (가) | 40% | 비교 조건이 (나), (다)의 재귀 호출 방향과 논리적으로 일관되어야 인정 |
| (나) | 30% | |
| (다) | 30% | |

부분 점수:

- **(나) 또는 (다)에서 `+1`/`-1` 오차로 틀린 경우 (예: `mid - 1` 자리에 `mid`)**: 해당 항목을 30%가 아닌 **10%**로 인정. 핵심 아이디어(어느 쪽 절반을 좁힐지)는 맞췄으나 종료/포함 범위 처리가 부정확한 경우.
- **그 외 틀린 경우 (재귀 호출 자체가 잘못됐거나 함수 인자 구조가 망가진 경우)**: 0%.

자주 나올 오답:

- (나)에 `binary_search(arr, target, low, mid)`처럼 `mid` 그대로 둠: `arr[mid]`는 이미 다른 값임이 확인된 상태이므로 다음 탐색에서 빼야 한다. 또한 `low == high`인 마지막 한 칸 상황에서 종료 보장이 깨져 무한 재귀가 발생. → 부분점수 10%.
- (가) 조건 방향과 (나)/(다) 재귀 호출 방향이 어긋남 (예: (가) `arr[mid] > target`인데 (나)에서 `mid + 1`부터 high까지 탐색): 논리 불일치이므로 (가) 0%, (나)·(다) 각각도 일관성 없으면 0%.

## 5번

### (1) 원인 모범 답안

이진 탐색은 리스트가 **정렬되어 있을 때**만 "절반을 버린다"는 판단이 성립한다. 그런데 `names`는 정렬되어 있지 않다.

실제 동작을 추적하면, 첫 호출에서 `mid = 4`, `names[4] = "윤예서"`. `"이효주" > "윤예서"`이므로 알고리즘은 오른쪽 절반(인덱스 5~9)을 탐색한다. 그러나 `"이효주"`는 실제로는 왼쪽 절반(인덱스 2)에 있다. 정렬 전제가 깨졌기 때문에 "절반을 버린다"는 판단이 틀린 방향을 가리켰고, 그 결과 타겟이 들어있는 절반을 통째로 버린 채 탐색이 진행되어 결국 `low > high`에 걸려 `-1`이 반환된다.

### (2) 수정 모범 답안

`binary_search` 호출 직전에 다음 한 줄을 추가한다.

```python
names.sort()
```

추가 위치: `result = binary_search("이효주", 0, len(names) - 1)` **바로 앞 줄**.

수정 후:

```python
names.sort()
result = binary_search("이효주", 0, len(names) - 1)
print(result)
```

파이썬 문자열은 유니코드 코드포인트 순서로 비교되며, 이 순서는 한글 사전 순서(가나다 순)와 일치하므로 정렬 후 이진 탐색이 올바르게 동작한다.

## 6번

채점: 코드에 한 군데라도 틀린 부분이 있으면 0%, 완전히 맞아야 100%. 부분 점수 없음.

모범 답안 코드 (해설 겸용):

```python
def sort(arr):
    n = len(arr)
    for i in range(n):
        for j in range(0, n - 1 - i):
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]
    return arr


print(sort([3, 1, 4, 2]))
```

## 7번

모범 답안 (간단형):

`b = a`는 리스트를 복사하는 게 아니라 같은 리스트 객체에 `b`라는 이름표를 하나 더 붙이는 연산이다. 이후 `a[1] = 99`는 그 공유된 객체의 내부 값을 바꾸는 것이므로(mutation), `b`로 조회해도 같은 객체이기에 변경된 결과 `[1, 99]`가 보인다.

## 8번

모범 답안 (간단형):

`b = a`까지는 7번과 동일하게 `a`와 `b`가 같은 `[1, 2]` 객체를 가리킨다. 하지만 `a = [3, 4]`는 객체 내부를 바꾸는 연산이 아니라, **`a` 이름표를 새로 만들어진 `[3, 4]` 객체로 옮기는 연산**이다(rebinding). `b`의 이름표는 원래 `[1, 2]` 객체에 그대로 붙어 있으므로, `print(a)`는 `[3, 4]`, `print(b)`는 `[1, 2]`가 출력된다.

## 9번

모범 답안:

`[0] * 3`은 `[0, 0, 0]` 리스트 하나를 만든다. 바깥의 `* 3`은 이 리스트를 세 개 새로 만드는 것이 아니라, **이미 만들어진 같은 리스트에 이름표 세 개**를 붙여 바깥 리스트에 담는다. 따라서 `grid[0]`, `grid[1]`, `grid[2]`는 모두 메모리상 동일한 하나의 리스트에 붙은 서로 다른 이름이다.

`grid[0][0] = 1`은 그 공유된 리스트의 0번 칸을 바꾸는 연산이므로, `grid[1][0]`과 `grid[2][0]`으로 조회해도 같은 변경이 그대로 보여 세 행이 모두 `[1, 0, 0]`이 된다.

def total_by_menu(sales):
    menu_count = len(sales[0])
    totals = [0] * menu_count
    for row in sales:
        for j in range(menu_count):

def find_top(scores):
    best_i = 0
    best_j = 0
    best_value = scores[0][0]
    for i in range(len(scores)):

## 19번

모범 답안 코드:

```python
def max_window_sum(data, k):
    best_idx = 0
    best_sum = sum(data[:k])
    for i in range(1, len(data) - k + 1):
        s = sum(data[i:i+k])
        if s > best_sum:
            best_sum = s
            best_idx = i
    return (best_idx, best_sum)
```

핵심:
- 시작 인덱스 상한은 `len(data) - k`이므로 `range(...)`의 끝값은 `len(data) - k + 1`. `+1`을 빠뜨리면 마지막 윈도우 누락.
- 비교 연산자 `>` (11번과 동일 이유로 `>=` 사용 시 동률에서 뒤쪽 위치가 반환됨).

## 20번

모범 답안 코드:

```python
def order(menu, budget):
    bought = []
    while True:
        m = input()
        if m == '끝':
            break
        if m not in menu:
            print("없는 메뉴")
            continue
        if menu[m] > budget:
            print("예산 부족")
            continue
        budget -= menu[m]
        bought.append(m)
        print(f"{m} 주문 완료 (잔액 {budget}원)")
    return (bought, budget)
```

핵심:
- 검사 순서는 반드시 `'끝'` → 메뉴 존재 여부(`m not in menu`) → 예산 충분 여부 → 정상 처리. 메뉴 존재 검사 전에 `menu[m]`을 조회하면 없는 메뉴 입력 시 `KeyError`.
- 분기에서 `break`가 아닌 `continue`로 다음 입력을 받아야 함. `break`로 잘못 쓰면 첫 잘못된 입력에서 함수가 종료됨.
- `bought = []`는 `while` 바깥. 안에 두면 매 반복마다 초기화.
