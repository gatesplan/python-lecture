# 26-2 정보수업 내용 보강 (260530)

주요내용에서 단순화한 부분의 심화 설명과, 수업 중 나온 학생 질문(김하은)에 대한 답변 정리.

## 1. IPv4와 IPv6 - 왜 바꾸나

- **IPv4 (32비트)**: 약 2^32 = 약 43억 개의 주소. 표기는 점으로 구분한 10진수 4자리 (예: `192.168.0.1`)
- 전 세계 기기(PC, 스마트폰, IoT 기기 등) 수가 43억을 넘어서며 **주소 고갈** 발생
- **IPv6 (128비트)**: 약 2^128 개(사실상 무한). 표기는 콜론으로 구분한 16진수 8그룹 (예: `2001:0db8:85a3:...`)

**교사 참고**: 학생들이 "2^32가 왜 부족하냐"고 물으면, 한 사람이 폰/노트북/태블릿/IoT 기기까지 여러 대를 쓰는 시대라는 점을 들면 직관적이다.

## 2. DNS 동작과 도메인 구조 정정

### DNS가 하는 일 (간단히)
1. 사용자가 도메인 이름을 입력 (예: `www.naver.com`)
2. DNS 서버에 "이 도메인의 IP가 뭐냐"고 질의
3. DNS가 대응하는 IP 주소를 회신
4. 그 IP로 실제 접속

비유: 전화번호부에서 이름으로 번호를 찾는 것과 같다.

### 필기 정정: www는 프로토콜이 아니다
필기에 `www -> 프로토콜`로 적혀 있으나 부정확하다.

- **프로토콜**: 통신 규약. `http`, `https` 등 (주소 맨 앞 `https://` 부분)
- **www**: 호스트 이름. 웹 서버를 가리키는 관례적 이름 (프로토콜이 아님)

전체 주소 구조: `프로토콜://호스트.도메인/경로`
- 예) `https://www.naver.com/news`
  - `https` = 프로토콜, `www` = 호스트 이름, `naver.com` = 도메인, `/news` = 경로

### 호스트란 무엇이고, www는 필수인가

**호스트(host)**: 네트워크에 연결되어 통신하는 개별 컴퓨터, 특히 서비스를 제공하는 **서버**. 주소에서 "어느 서버에 접속할지" 가리키는 부분이다.

여기서 "호스트"는 두 가지 뜻으로 쓰여 헷갈리기 쉽다.

| 구분 | 가리키는 것 | 필수 여부 |
|---|---|---|
| 넓은 의미 (접속할 서버 주소) | `www.naver.com` 전체 | **필수** |
| 좁은 의미 (호스트 이름 = 서브도메인 라벨) | 도메인 앞에 붙는 `www`, `mail`, `blog` | 선택 |

**www는 필수가 아니다.**
- `www`는 옛날에 한 기관이 여러 서버를 나눠 운영하며 웹 서버에 붙이던 관례적 이름이다 (예: `www`=웹, `ftp`=파일, `mail`=메일 -> `www.example.com`, `ftp.example.com`)
- 요즘은 `github.com`처럼 www 없이 접속되는 사이트가 많다. `naver.com`만 쳐도 열리는 건 도메인 주인이 DNS/리다이렉트를 그렇게 설정해 둔 것이다
- 즉 `www`(라벨)는 빼도 된다

**하지만 호스트(접속할 서버를 가리키는 부분)는 필수다.**
- 빠질 수 있는 건 `www` 라벨이지, 서버를 가리키는 주소 자체가 아니다
- 어떤 형태로든 서버를 지목해야 한다: `naver.com`도 호스트, `www.naver.com`도 호스트, IP 주소(예: `223.130.x.x`)도 호스트가 된다
- 호스트가 아예 없으면 "어느 서버로 가라"는 정보가 없어 접속이 불가능하다

한 줄 정리: **`www`는 선택(빼도 됨), 호스트(서버를 가리키는 부분)는 필수.** "URL에서 호스트가 불필요해 보이는" 건 `www`를 호스트와 같은 것으로 본 데서 온 착각이다.

## 3. 유선 vs 무선 - 비교 보충

| 항목 | 유선 | 무선 |
|---|---|---|
| 속도 | 빠르고 안정적 | 환경에 따라 변동 |
| 신뢰성 | 높음 (간섭 적음) | 신호 간섭/거리 제한 있음 |
| 이동성 | 낮음 (선에 묶임) | 높음 (자유로운 이동) |
| 보안 | 상대적으로 안전 | 도청 등에 취약 -> 암호화 필요 |
| 설치 | 선 정리가 복잡 | 간편 |

## 4. IoT 네트워크 기술 비교 (NFC / RFID / Beacon / ZigBee / WiFi)

IoT의 N(네트워크) 요소에 쓰이는 대표 통신 기술들. 각자 거리, 속도, 전력 소모, 용도가 다르다.

| 기술 | 통신 거리(대략) | 전송 속도 | 전력 소모 | 대표 용도 |
|---|---|---|---|---|
| NFC | ~10cm | 매우 느림 | 매우 낮음 | 교통카드, 모바일 결제, 태그 인식 |
| RFID | 수 cm ~ 수 m | 느림 | 낮음 (수동형은 자체 전원 없음) | 출입증, 물류/재고 관리 |
| Beacon (BLE) | ~수십 m | 느림 | 매우 낮음 | 실내 위치 안내, 근접 알림 |
| ZigBee | ~10 ~ 100 m | 느림 (~250 kbps) | 낮음 | 스마트홈 센서 네트워크 |
| WiFi | ~수십 m (실내) | 빠름 (수십 Mbps ~ 기가급) | 높음 | 인터넷 연결, 대용량 데이터(영상/파일) 전송 |

### 김하은 학생 질문 답변

> 질문: "네트워크 부분 예시 그래프에서 와이파이(WiFi)가 제일 높았는데, 와이파이가 그중 어떤 부분에서 뛰어난가요?"

그래프에서 WiFi가 가장 높았다면 그 축은 **전송 속도(대역폭)** 일 가능성이 크다. WiFi의 강점은 세 가지다.

1. **전송 속도/대역폭이 압도적으로 크다.** 다른 기술들은 작은 신호나 짧은 데이터를 주고받는 수준이지만, WiFi는 수십 Mbps에서 기가급까지 가능해 영상이나 큰 파일도 전송할 수 있다.
2. **통신 거리가 실내 전역을 덮을 만큼 충분하다.** NFC(약 10cm), RFID(수 m)에 비해 훨씬 넓다.
3. **인터넷에 바로 연결된다.** 가정/학교의 인터넷 공유기에 그대로 붙어 별도 장치 없이 인터넷과 직접 통신한다.

**단, WiFi가 항상 최고인 것은 아니다 (트레이드오프).** WiFi는 **전력 소모가 크다.** 그래서 배터리로 오래 동작해야 하는 작은 센서에는 부적합하고, 그런 경우에는 저전력인 ZigBee나 Beacon이 더 유리하다. NFC/RFID는 초근거리에서 태그를 인식하거나 결제하는 데 특화되어 있다.

한 줄 정리: **"빠르고 멀리, 인터넷에 바로" = WiFi / "느려도 오래(저전력), 가까이" = ZigBee, Beacon, NFC**

## 5. 로컬 공유 보안 4원칙 - 왜 필요한가

| 원칙 | 이유 |
|---|---|
| 최소 공유 | 공유 범위가 넓을수록 정보가 샐 위험이 커짐 |
| 사전 검사 | 악성 파일이나 엉뚱한 대상에게 보내는 실수를 막음 |
| 정보 보호 | 개인정보/민감 자료가 제3자에게 노출되지 않게 함 |
| 권한 설정 | 읽기/쓰기 권한을 제한해 무단 수정/삭제를 방지함 |
